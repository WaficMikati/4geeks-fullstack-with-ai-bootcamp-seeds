## Section 02: Exception Handling Fundamentals

### 02.0 Try/Catch/Finally Pattern in Python 📖

**Learning Objectives:**
- Implement the try/except/finally pattern correctly in Python
- Understand when to use specific exception handling vs. general exception catching
- Design exception handling that maintains application stability

**Content Outline:**
- Basic try/except syntax and flow control
- Catching specific exceptions vs. general Exception class
- The finally block: guaranteed cleanup code
- Multiple except blocks for different error types
- Best practices for exception handling scope

**Transitions:**
With the fundamentals of try/except/finally understood, let's practice building robust exception handlers that handle real-world scenarios gracefully.

**Key Principles:**
- **Scope Specific**: Use try/except specifically where dangerous operations occur, not around entire functions
- **Finally for cleanup**: Use finally blocks to ensure resources are cleaned up regardless of success or failure
- **Specific over general**: Catch specific exceptions when possible rather than broad Exception catching
- **Fail gracefully**: Applications should handle errors without crashing

**Examples:**
- File operations with proper cleanup using finally
- API calls with specific handling for network vs. data errors
- Database operations with connection cleanup

**Anti-patterns (woven in):**
- **Wrapping Gigante**: Putting entire applications or large functions in one massive try/catch instead of targeting specific risky operations
- **Silent failures**: Catching exceptions but not logging them or informing the user anything went wrong
- **Exception as control flow**: Using try/catch for normal program logic instead of just error handling

### 02.1 Building Robust Exception Handlers 💻

**Challenge Prompt:**
Build a file processing function that handles multiple types of errors gracefully and always cleans up resources, demonstrating proper exception handling patterns.

**Solution Code:**
```python
def process_user_file(filename):
    file_handle = None
    try:
        print(f"Opening file: {filename}")
        file_handle = open(filename, 'r')
        
        content = file_handle.read()
        
        # Process the content (might raise ValueError)
        lines = content.split('\n')
        numbers = [int(line.strip()) for line in lines if line.strip()]
        
        total = sum(numbers)
        print(f"Successfully processed {len(numbers)} numbers. Total: {total}")
        return total
        
    except FileNotFoundError:
        print(f"Error: Could not find file '{filename}'. Please check the file path.")
        return None
    except ValueError as e:
        print(f"Error: File contains invalid data. {e}")
        return None
    except PermissionError:
        print(f"Error: Permission denied accessing '{filename}'.")
        return None
    finally:
        if file_handle:
            file_handle.close()
            print("File closed successfully.")

# Test the function
if __name__ == "__main__":
    result = process_user_file("numbers.txt")
```

## Section 03: Debugging Techniques and Tools

### 03.0 Print Debugging vs Python Debugger (pdb) 📖

**Learning Objectives:**
- Compare print debugging with systematic debugging tools
- Understand when to use print statements vs. interactive debugging
- Learn the basic commands and workflow of Python's pdb debugger

**Content Outline:**
- Print debugging: quick insights but limited scope
- Python debugger (pdb): interactive investigation of program state
- Setting breakpoints with pdb.set_trace()
- Basic pdb commands: step, next, continue, print variables
- When each approach is most effective

**Transitions:**
Now let's put these debugging techniques into practice with an interactive debugging session where you'll solve a real bug using pdb.

**Key Principles:**
- **Evidence over assumption**: Use debugging tools to see actual program state rather than guessing what's happening
- **Systematic investigation**: Follow a consistent process to isolate problems
- **Right tool for the job**: Print debugging for quick checks, pdb for complex issues
- **Interactive exploration**: Debugging tools let you examine variables and execution flow step by step

**Examples:**
- Using print statements to track variable values through a calculation
- Using pdb to step through a loop that's producing unexpected results
- Setting breakpoints to examine the state when an exception occurs

**Patterns (woven in):**
- **Three-state debugging**: Identify where the bug occurs (before/during/after a specific operation), then focus investigation on that area
- **Binary search debugging**: Add debugging at the middle of a problematic section, then narrow down based on whether the issue is before or after that point

### 03.1 Interactive Debugging Session 💻

**Challenge Prompt:**
Debug a function that calculates student grades but produces incorrect results, using Python's pdb debugger to identify and fix the logical error.

**Solution Code:**
```python
import pdb

def calculate_final_grades(students):
    final_grades = {}
    
    for student_name, scores in students.items():
        # Bug was here: didn't exclude the lowest score
        # Original: total = sum(scores)
        
        # Fixed: Remove lowest score before calculating average
        if len(scores) > 1:
            working_scores = scores.copy()
            working_scores.remove(min(working_scores))
        else:
            working_scores = scores
            
        total = sum(working_scores)
        average = total / len(working_scores)
        
        # Assign letter grade
        if average >= 90:
            letter_grade = 'A'
        elif average >= 80:
            letter_grade = 'B'
        elif average >= 70:
            letter_grade = 'C'
        else:
            letter_grade = 'F'
            
        final_grades[student_name] = {
            'average': round(average, 2),
            'letter': letter_grade
        }
    
    return final_grades

# Test data
```

## Section 04: Assessment and Application

### 04.0 Error Communication and User Experience 📖

**Learning Objectives:**
- Design error messages that help users rather than confuse them
- Implement user-friendly error handling in applications
- Balance technical accuracy with user comprehension in error communication

**Content Outline:**
- Translating technical errors into user-friendly messages
- Providing actionable next steps in error messages
- UI patterns for error states: pending, fulfilled, rejected
- When to log technical details vs. show user-friendly messages
- Call-to-action buttons in error scenarios (retry, contact support, go back)

**Transitions:**
Let's now assess your complete understanding of Python debugging through a comprehensive evaluation.

**Key Principles:**
- **Human translation**: Convert technical errors into language users understand
- **Actionable guidance**: Always provide users with clear next steps
- **Graceful degradation**: Show helpful error states instead of broken interfaces  
- **Appropriate detail**: Log technical information for developers, show friendly messages to users

**Examples:**
- Instead of "KeyError: 'user_id'" show "We couldn't find your account information. Please try logging in again."
- Instead of "ConnectionError" show "We're having trouble connecting to our servers. Please check your internet connection and try again."
- Error pages with "Try Again" buttons instead of dead ends

**Patterns (woven in):**
- **UI of three states**: Pending (show spinner), Fulfilled (show data), Rejected (show helpful error message)
- **Call to Action (CTA)**: Every error message includes a clear action the user can take
- **Finally pattern**: Use finally blocks to ensure loading spinners disappear whether requests succeed or fail

**Anti-patterns (woven in):**
- **Technical dumping**: Showing users raw error messages like "Error 500: Unexpected token < in JSON at position 0"  
- **Dead end errors**: Error messages that just say "Error" without any guidance on what to do next
- **Exposed sensitive data**: Accidentally revealing internal system information in error messages during debugging
